- **Remove chat panel in favor of terminal-based Pi**: Remove the Flutter chat panel entirely and have users invoke `pi` directly from the terminal pane. The backend would need to spawn `pi` in interactive mode (not RPC) via `docker exec`, similar to how the shell terminal works. Requires `terminal_manager.py` to support multiple concurrent terminal sessions per workspace (one for Pi, one for shell). See CLI.md "Broader Direction" section.

## Backend

- **Update plugin docs to reflect new architecture**: "Bark plugins" are now just normal Pi extensions (npm packages with `pi.registerTool()`) that optionally include a `bark/` subdirectory with Dart code for browser-side actions. Update ARCHITECTURE.md plugin section, remove references to AG-UI `HOST_TOOL_REQUEST` delegation, and document the browser bridge flow (`@bark/bridge` → backend → WebSocket → Flutter plugin registry).
- **Handle unsupported extension_ui_request methods**: Pi extensions like pi-subagents send `extension_ui_request` with methods the backend doesn't handle (e.g., `setWidget` with `widgetKey: "subagent-async"`). Pi blocks waiting for a response that never comes, causing a timeout. Either respond with an error/empty value so Pi can continue, or filter out extensions that rely on TUI-only features.
- **Hosted app URLs should respect external headers**: The `get_hosted_url` tool generates URLs using `BARK_HOSTING_*` env vars passed to the container. These are derived from `X-Forwarded-Host`/`X-Forwarded-Proto` headers at WebSocket connect time, but if the hosting environment changes (e.g., different reverse proxy), the container's cached values become stale. Consider re-deriving hosting info on each `get_hosted_url` call or providing a mechanism to update the container's env vars without restart.
- **Idle timeout for heartbeat-only sessions**: Heartbeats prevent idle timeout, but a container kept alive solely by heartbeats (no user input, terminal output, or Pi events for an extended period) should eventually be stopped. Consider a two-tier timeout: a short one (e.g., 30 min) based on real activity, and a longer one (e.g., 4 hours) that heartbeats alone cannot extend.
- **Per-user containers for shared workspaces**: Currently each workspace has one container shared by all connections. If two people open the same workspace (via the Flutter UI or `bark ws shell`), they share a single Pi instance and shell. Consider spawning a separate container per user per workspace so users don't interfere with each other's Pi sessions or terminal state. Trade-offs: higher resource usage, need to sync filesystem changes between containers (or use a shared volume), port allocation becomes per-container instead of per-workspace. **Known hazard with shared containers**: the connection refcount in `cleanup_connection` (ws_handler.py line 937) kills the container when the last WebSocket disconnects (`remaining == 0`). If multiple clients share a workspace and one client's connections close (e.g., browser tab closed, network glitch), the refcount can drop to zero and kill the container while another client is still actively using it. This caused a persistent CI test failure — the fanout E2E tests shared a single workspace across all tests, and previous tests' WebSocket disconnects would race with the next test's connections, killing the container mid-prompt. Fixed by giving each test its own workspace, but the underlying issue remains for real multi-user scenarios.
- **SQLite migration tooling**: Research SQLite migration tools (e.g., Alembic, yoyo-migrations) for when dropping and recreating the database is no longer feasible. Currently schema changes are handled by `CREATE TABLE IF NOT EXISTS` with manual `ALTER TABLE` fallbacks, which won't scale as the schema evolves.
- **Evaluate removing AG-UI and pi_rpc_client**: If the chat panel is removed in favor of terminal-based Pi, the AG-UI translation layer (`agui_translator.py`) and Pi RPC client (`pi_rpc_client.py`) become unnecessary. Discuss whether to remove them entirely or keep them as an optional mode. Removing simplifies the backend significantly but loses the ability to render structured tool output in the UI.
- **Clean up stale FIFO stderr warning in pi_rpc_client**: The `read_loop` logs a warning about "expected ENOENT for settings.json FIFO after startup read" but we no longer use a FIFO for settings.json. Update or remove the warning message.
- **Rename user_store.py**: `user_store.py` contains all database access — users, workspaces, messages, ports, roles, login attempts. Rename to `db.py` or `model.py` to better reflect its scope.
- **Remove SOLIPLEX_URL references**: Soliplex integration is handled by a plugin, but the backend still passes `SOLIPLEX_URL` as an env var to containers. Remove the references from `container_manager.py` and related code.
- **Bark CLI (`bark ws shell`)**: Phase 1 done — CLI tool for accessing Bark containers from the terminal. Connects to the Bark backend over HTTP + WebSocket, drops into a bash shell inside a workspace container. Works locally and against remote Bark servers. See CLI.md for future phases (host path mounting, SSH access, custom Docker images, file sync).
- **Bark plugins as Pi extensions**: Bark plugins should just be normal Pi extensions (npm packages with `package.json` and a `"pi"` section). If a plugin also contains Dart code (e.g., a `lib/` directory or `pubspec.yaml`), the build detects it and treats it specially — compiling the Dart and wiring up client-side UI. This eliminates the custom `plugin.yaml` / `extension.ts` format and aligns with the Pi extension ecosystem directly.

- **Test SMTP email delivery**: The email service has unit tests with mocked SMTP, but no integration test for real SMTP delivery. - **Container resource limits**: Add CPU/memory limits to containers to prevent runaway processes.

## Docker

- **Preserve user Pi settings across container restarts**: The entrypoint wipes `~/.pi/agent/` and overwrites `settings.json` on every container start, losing any user customizations (e.g., settings Pi wrote during the session). Pi has a two-level settings system: global (`$PI_CODING_AGENT_DIR/settings.json`) and project (`<cwd>/.pi/settings.json`), where project settings deep-merge on top of global and take precedence. User customizations in `/work/.pi/settings.json` would survive since `/work` is persistent. Also consider switching from `pi install` to `npm install -g` for builtin extensions to avoid the `"packages"` key in settings.json entirely (Pi discovers globally installed extensions without it). User-installed extensions (`pi install` during a session) would be lost on restart either way since `~/.pi/agent/` is wiped.
- **Lock npm extension versions**: The `pi install npm:pi-subagents` commands in the Dockerfile install whatever `latest` resolves to at build time, with no lock file. Pin to specific versions (e.g., `pi install npm:pi-subagents@0.25.0`) or find a way to lock versions so builds are reproducible.

## Testing

- **E2E: use Flutter semantics tree for testability**: Enabling `RendererBinding.instance.ensureSemantics()` exposes real DOM elements (`<input aria-label="Email">`, `<flt-semantics role="button">Login</flt-semantics>`) that Playwright can interact with via standard selectors instead of coordinate clicks. However, the semantics overlay intercepts keyboard events globally, which breaks xterm.dart (terminal input never reaches the canvas). Investigated `ExcludeSemantics` on the terminal widget and on-demand enable/disable via the "Enable accessibility" button — both still break the terminal. Need a way to selectively disable the semantics overlay for the terminal area, or find an alternative approach to make Flutter canvas apps testable without coordinates.
- **E2E: use UI instead of API for setup/teardown**: E2E tests currently use the API directly to upload files, delete workspaces, etc. These should use the UI instead to better test real user flows and catch UI-level regressions.
- **E2E: replace message polling with WebSocket events**: LLM E2E tests poll the messages REST API to detect agent completion. Instead, open a WebSocket in the test (or intercept the page's WebSocket via `page.on('websocket')`) and listen for `RUN_FINISHED` events. This would be faster, more reliable, and avoid arbitrary timeouts.
